# Testing & Release Flow

How to develop, test, and publish new versions of `homebridge-grohe-sense`.

## Project facts

- **Module format:** native ESM (`"type": "module"`). Relative imports must include `.js` extensions.
- **Supported runtime:** Node `^22.12.0 || ^24.0.0`, Homebridge `^1.6.0 || ^2.0.0`.
- **Build output:** `dist/` (git-ignored, never committed). It is generated by `tsc` and only exists in the published npm tarball.
- **Published files:** `dist/`, `config.schema.json`, `icons/`, `img/` (see the `files` allowlist in `package.json`), plus the auto-included `README.md`, `LICENSE`, `package.json`.

## One-time setup

- **npm auth (for CI publishing):** uses npm **Trusted Publishing (OIDC)** - no `NPM_TOKEN` secret. Configure the trusted publisher once on npmjs.com (package → Settings → Trusted Publishing) pointing at this repo's `.github/workflows/publish.yml` workflow on a GitHub-hosted runner. The workflow has `id-token: write` and upgrades to the latest npm (trusted publishing needs npm >= 11.5.1).
- **npm auth (for manual publishing):** `npm login`.
- **SSH for git push:** the `origin` remote uses SSH (`git@github.com:faune/homebridge-grohe-sense.git`); your key must be registered on GitHub.

## Everyday development

```bash
npm install          # install deps (uses package-lock.json)
npm run build        # rimraf dist + tsc
npm run lint         # eslint . (flat config in eslint.config.js)
```

Quick runtime sanity check that the built ESM module loads:

```bash
node --input-type=module -e "import('./dist/index.js').then(m => console.log('ok', typeof m.default))"
```

> Note: use Node 22 or 24. On Node 25+ a transitive dependency of `fakegato-history`
> (`googleapis` → `buffer-equal-constant-time`) crashes because `SlowBuffer` was removed.

## Testing an unreleased build on a Raspberry Pi

Because `dist/` is not committed, install a packed tarball (it contains the build) rather than installing from git.

On your dev machine:

```bash
npm run build
npm pack                                   # produces homebridge-grohe-sense-<version>.tgz
scp homebridge-grohe-sense-*.tgz pi@raspberrypi.local:/tmp/
```

On the Pi:

```bash
node -v                                    # confirm v22.x or v24.x (else: sudo hb-service update-node 22)
sudo npm install -g /tmp/homebridge-grohe-sense-*.tgz
sudo hb-service restart
```

**Recommended:** run the plugin in a **Child Bridge** (Homebridge UI → Ondus plugin →
Bridge Settings) while testing, so a failure can't take down the whole instance.
Watch logs with `sudo hb-service logs` or the UI Logs tab.

Revert after testing:

```bash
sudo npm install -g homebridge-grohe-sense@latest   # back to the published release
# or: sudo npm uninstall -g homebridge-grohe-sense
sudo hb-service restart
```

## Releasing a new version

Releases are published by the `publish.yml` GitHub Action when a **GitHub Release**
is published. `npm publish` runs `prepublishOnly` (lint + build) automatically, so the
tarball is always freshly built and linted.

1. Bump the version (updates `package.json`; `PLUGIN_VERSION` is read from it at runtime):

   ```bash
   npm version patch    # or: minor | major | 2.1.0
   ```

   `npm version` also creates a `vX.Y.Z` git tag and commit.

2. Push the commit and tag:

   ```bash
   git push && git push --tags
   ```

   This triggers `build.yml` (lint + build on Node 22 & 24).

3. Create the GitHub Release for the tag (UI → Releases → Draft a new release →
   pick the `vX.Y.Z` tag → add notes → Publish). Publishing the release triggers
   `publish.yml`, which runs `npm publish` and uploads to npm under the `latest` tag.

4. Verify: `npm view homebridge-grohe-sense version`.

### Beta / pre-release

Mark the GitHub Release as a **pre-release**. `publish.yml` then publishes under the
`beta` dist-tag instead of `latest`, so it won't be installed by default.

Use a pre-release version string so npm treats it as a beta:

```bash
npm version 2.1.0-beta.0
git push && git push --tags
# create the GitHub Release, check "Set as a pre-release"
```

Install a beta on the Pi for testing: `sudo npm install -g homebridge-grohe-sense@beta`.

### Manual publish (fallback, if not using the Action)

```bash
npm publish               # prepublishOnly runs lint + build first
npm publish --tag beta    # for a pre-release version
```

## Pre-release checklist

- [ ] `npm run build` and `npm run lint` pass locally.
- [ ] `npm audit` is clean (production must be 0: `npm audit --omit=dev`).
- [ ] Version bumped in `package.json`.
- [ ] Tested on a real Homebridge v2 instance (ideally in a child bridge).
- [ ] `engines` still reflects supported Node/Homebridge ranges.
